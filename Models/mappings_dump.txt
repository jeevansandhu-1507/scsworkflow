SCS CLIENT FUNDING → GL MAPPING SPEC (ZOHO IMPLEMENTATION)
Purpose
Define deterministic GL mapping for:
Funding Commitments (Forecasting)
Invoicing (Actuals)
1. CORE PRINCIPLES
Funding Type = Primary Driver
Plan Type = Commitment Anchor ONLY
Placement Type = ONLY when Funding Status = Permanent 
Placement Type can be found in Client (Project) Record
One row per GL (lookup-driven, no free logic)
No partial matches allowed
2. DATA SOURCES (LOOKUP TABLES)
Table A: Funding_Commitment_GL_Mapping
Used for: Commitment / Forecast
Fields:
Funding_Type
Plan_Type
Funding_Status (Permanent / Temporary)
Support_Type
Placement_Type (nullable)
GL_Code
GL_Name
Account_ID
Derivation_Rule
Is_Placement_Applicable (Y/N)
Table B: Invoicing_GL_Mapping
Used for: Billing / Actuals
Fields:
Funding_Type
Funding_Status
Support_Type
Placement_Type (nullable)
GL_Code
GL_Name
Account_ID
Derivation_Rule
3. FIELD BEHAVIOUR (ZOHO UI)
Required Order
Funding Type (required FIRST)
Funding Status (auto or selected)
Plan Type (commitments only)
Support Type
Placement Type (conditional)
Conditional Logic
IF Funding_Status != "Permanent" → hide Placement_Type → set Placement_Type = null
IF Module = Invoicing → hide Plan_Type
IF Plan_Type = "Restricted" → filter GLs to restricted-compatible only
4. GL DERIVATION LOGIC
4.1 FUNDING COMMITMENT (FORECAST)
Match in this order:
GL = lookup where: Funding_Type AND Plan_Type AND Funding_Status AND Support_Type AND (Placement_Type if applicable) 
4.2 INVOICING (ACTUALS)
Match in this order:
GL = lookup where: Funding_Type AND Funding_Status AND Support_Type AND (Placement_Type if applicable) 
5. ZOHO DELUGE LOGIC (PSEUDO CODE)
🔹 Commitment GL Mapping
criteria = "Funding_Type == \"" + input.Funding_Type + "\"" criteria += " && Funding_Status == \"" + input.Funding_Status + "\"" criteria += " && Support_Type == \"" + input.Support_Type + "\"" if(input.Plan_Type != null) { criteria += " && Plan_Type == \"" + input.Plan_Type + "\"" } if(input.Funding_Status == "Permanent" && input.Placement_Type != null) { criteria += " && Placement_Type == \"" + input.Placement_Type + "\"" } record = Funding_Commitment_GL_Mapping[criteria] if(record.isEmpty()) { alert("No valid GL mapping found. Check funding configuration.") cancel submit } input.GL_Code = record.GL_Code input.Account_ID = record.Account_ID 
🔹 Invoicing GL Mapping
criteria = "Funding_Type == \"" + input.Funding_Type + "\"" criteria += " && Funding_Status == \"" + input.Funding_Status + "\"" criteria += " && Support_Type == \"" + input.Support_Type + "\"" if(input.Funding_Status == "Permanent" && input.Placement_Type != null) { criteria += " && Placement_Type == \"" + input.Placement_Type + "\"" } record = Invoicing_GL_Mapping[criteria] if(record.isEmpty()) { alert("No valid GL mapping found for invoice.") cancel submit } input.GL_Code = record.GL_Code input.Account_ID = record.Account_ID 
6. VALIDATION RULES
Hard Stops
No match → block transaction
Funding Type missing → block
Invalid combination → block
Controls
Prevent cross-funding:
Passport → only Passport GLs
MCCSS → only MCCSS GLs
Placement enforcement:
Only allowed if Permanent
Legacy GLs:
Allow = false (new entries)
Allow = true (historical edits only)
7. SPECIAL HANDLING
Unrestricted Planning Accounts (5900–5965)
Used ONLY in commitments
NEVER used in invoicing
Multi-Funding Clients
Each record must tie to ONE Funding Type
No mixed funding in single transaction
8. EXPECTED OUTCOME
Deterministic GL assignment ✅
Clean audit trail ✅
Accurate forecast vs actual reporting ✅
No manual GL selection required ✅
